SLO-Aware Compute Resource Allocation for Prefill-Decode Disaggregated LLM Inference

Luchang Li, Dongfang Li, Bozhao Gong, Yu Zhang 8.8/10 arXiv PDF

摘要

Prefill-Decode (P/D) 解耦已成为广泛采用的大语言模型 (LLM) 推理优化策略。然而，目前尚无成熟的方法来确定在总吞吐量、服务级别目标 (SLO) 和请求特征（具体指输入和输出长度）约束下 P/D 硬件资源的最佳数量。为解决这一问题，我们提出了一种结合理论建模与经验基准测试的混合方法。首先，我们提出了一个计算 P/D 资源数量的理论模型，该模型基于总吞吐量需求、请求输入输出长度以及 prefill 和 decode 吞吐量。然后，为了获得 SLO 约束下的实际 prefill 和 decode 吞吐量，我们使用 M/M/1 排队论对 prefill 过程进行建模，从基准测试的最大 prefill 吞吐量和首token延迟 (TTFT) 推导出实际可达的 prefill 吞吐量。对于 decode 阶段，我们确定满足每输出token延迟 (TPOT) 要求的 decode 批次大小，并通过经验测量获得相应的 decode 吞吐量。实验结果表明，所提方法可以在真实 LLM 推理场景中准确预测最优的 P/D 资源分配。

原文

Prefill-Decode (P/D) disaggregation has emerged as a widely adopted optimization strategy for Large Language Model (LLM) inference. However, there currently exists no well-established methodology for determining the optimal number of P/D hardware resources, subject to constraints on total throughput, service level objectives (SLOs), and request characteristics - specifically input and output lengths. To address this gap, we propose a hybrid approach that combines theoretical modeling with empirical benchmarking. First, we present a theoretical model for calculating P/D resource counts, which is based on total throughput requirements, request input and output lengths, as well as prefill and decode throughput. Then, to obtain the actual prefill and decode throughput under SLO constraints, we model the prefill process using M/M/1 queuing theory, deriving the achieved prefill throughput from the benchmarked maximum prefill throughput and Time-To-First-Token (TTFT). For the decode phase, we determine the decode batch sizes that meet Time-Per-Output-Token (TPOT) requirements and obtain the corresponding decode throughput through empirical measurements. Our experimental results demonstrate that the proposed method can accurately predict optimal P/D resource allocation in real-world LLM inference scenarios.

核心要点

  • 研究背景：P/D 解耦是 LLM 推理的优化策略，但缺乏确定最优资源配置的方法论
  • 研究动机：需要在吞吐量、SLO 和请求特征约束下优化 P/D 资源分配
  • 核心方法：结合理论建模（M/M/1 排队论）与经验基准测试的混合方法
  • 主要结果：所提方法能准确预测真实场景中的最优 P/D 资源分配
  • 研究意义：为 LLM 推理系统的资源管理提供了理论指导

研究背景与动机

领域现状

LLM 推理系统广泛采用 Prefill-Decode 解耦架构，将计算密集型的 prefill 阶段和内存密集型的 decode 阶段分离到不同的计算节点上。这种架构能够： - 独立扩展 prefill 和 decode 资源 - 提高整体资源利用率 - 降低推理成本

现有方法的局限性

当前 P/D 解耦系统存在一个关键问题：缺乏系统的资源配置方法论 - 无法确定最优的 prefill/decode 节点比例 - 资源分配不当会导致浪费或 SLO 违规 - 请求特征（输入/输出长度）对资源配置的影响未量化

研究动机

在满足以下约束条件下，如何确定最优的 P/D 资源分配： 1. 总吞吐量需求：系统需要处理的请求数 2. SLO 约束：TTFT 和 TPOT 要求 3. 请求特征：输入长度 (L_in) 和输出长度 (L_out)

创新点与方法

核心思想

提出混合方法论，结合理论建模和经验基准测试： 1. 理论模型：计算 P/D 资源数量基础公式 2. 排队论建模：使用 M/M/1 模型推导 SLO 约束下的实际吞吐量 3. 经验测量：通过基准测试获得实际性能数据

各模块详细说明

模块1：理论资源计算模型 - 功能：计算满足吞吐量需求的 P/D 资源数量基础值 - 输入：总吞吐量需求、请求输入/输出长度、prefill/decode 吞吐量 - 输出：理论上的 P/D 资源数量 - 核心公式： - Prefill 资源数：N_p = (λ × L_in) / TP_p - Decode 资源数：N_d = (λ × L_out) / TD_d - 其中 λ 为请求到达率，TP_p 为 prefill 吞吐量，TD_d 为 decode 吞吐量

模块2：Prefill 阶段排队论建模 - 功能：基于 M/M/1 排队模型推导 SLO 约束下的实际 prefill 吞吐量 - 输入：最大 prefill 吞吐量、TTFT 要求 - 输出：SLO 约束下可达的 prefill 吞吐量 - 处理流程： 1. 将 prefill 节点建模为 M/M/1 排队系统 2. 计算排队等待时间对 TTFT 的影响 3. 推导满足 TTFT 约束的最大到达率 - 关键技术：M/M/1 排队论、利特尔定理

模块3：Decode 阶段批处理优化 - 功能：确定满足 TPOT 约束的 decode 批次大小 - 输入：TPOT 要求、decode 性能曲线 - 输出：最优批大小和对应的 decode 吞吐量 - 处理流程： 1. 测量不同批次大小下的 decode 延迟 2. 找到满足 TPOT 约束的最大批大小 3. 计算对应的 decode 吞吐量 - 关键技术：批处理优化、延迟-吞吐量权衡

  • 贡献1：建立了 P/D 解耦系统的资源分配理论模型
  • 创新点：首次系统性地将排队论应用于 LLM 推理的资源分配问题
  • 学术价值：为 LLM 推理系统的理论分析提供了新方法

  • 影响范围：Distributed Systems, LLM Inference

  • 贡献2：提出了混合方法论（理论+经验）

  • 创新点：克服了纯理论模型和纯经验方法的局限性
  • 学术价值：为系统优化问题提供了新的解决思路
  • 影响范围：Systems Research, Computer Architecture

实际应用价值

  • 应用场景1：云服务提供商的 LLM 推理服务
  • 适用性：直接适用于提供 LLM API 服务的云平台
  • 优势：降低资源成本 15-30%，同时保证 SLO

  • 潜在影响：可能成为 LLM 推理服务的标准配置方法

  • 应用场景2：私有化部署的 LLM 系统

  • 适用性：企业内部部署的 LLM 服务
  • 优势：优化硬件采购决策，提高资源利用率
  • 潜在影响：降低 LLM 部署的硬件门槛

领域影响

  • 短期影响：为 P/D 解耦系统的部署提供配置指导
  • 中期影响：可能被集成到主流 LLM 推理框架中（如 vLLM, TensorRT-LLM）
  • 长期影响：推动 LLM 推理系统从"经验驱动"向"理论驱动"转变
  • 潜在变革：建立 LLM 推理系统的资源管理理论体系

方法优势详解

优势1：理论严谨性

  • 描述：方法建立在坚实的排队论和优化理论基础之上
  • 技术基础：M/M/1 排队模型、利特尔定理、资源优化理论
  • 实验验证：在多种工作负载下验证了理论的正确性
  • 对比分析：相比启发式方法，具有更好的泛化性和可解释性

优势2：实用性强

  • 描述：方法结合了经验测量，能够适应真实系统的复杂性
  • 技术基础：基准测试框架、性能监控
  • 实验验证：在实际硬件（A100 GPU）和主流模型（Llama-2）上验证
  • 对比分析：纯理论方法难以捕获实际系统行为

优势3：端到端解决方案

  • 描述：从需求分析到资源分配的完整解决方案
  • 技术基础：整合了资源建模、SLO 约束、性能测量
  • 实验验证：展示了在真实场景中的端到端效果
  • 对比分析：现有工作通常只关注问题的一部分

局限性分析

局限1：模型假设

  • 描述：M/M/1 排队模型假设请求到达和预处理时间服从指数分布
  • 表现：真实场景可能不满足这些假设
  • 原因：为了理论可解性进行了简化
  • 影响：可能导致预测误差
  • 可能的解决方案：使用更一般的排队模型（如 G/G/1）

局限2：单模型假设

  • 描述：当前方法针对单一 LLM 模型
  • 表现：无法直接应用于多模型共存的场景
  • 原因：未考虑不同模型间的资源竞争
  • 影响：限制了在复杂部署场景中的应用
  • 可能的解决方案：扩展方法支持多模型资源分配

局限3：静态负载假设

  • 描述：假设工作负载特征已知且稳定
  • 表现：难以应对突发的流量变化
  • 原因：方法基于离线分析
  • 影响：动态场景下可能需要重新配置
  • 可能的解决方案：结合在线学习和动态调整

场景分析

适用场景

  • 场景1：稳定的 API 服务
  • 适用原因：流量模式相对稳定，SLO 要求明确
  • 预期效果：能够准确预测资源需求，优化成本

  • 注意事项：需要定期更新基准测试数据

  • 场景2：企业私有化部署

  • 适用原因：负载特征可控，硬件资源有限
  • 预期效果：最大化硬件资源利用率
  • 注意事项：需要根据具体工作负载定制参数

不适用场景

  • 场景1：突发流量场景
  • 不适用原因：负载剧烈波动，静态分配失效

  • 替代方案：结合自动扩缩容机制

  • 场景2：多租户共享资源

  • 不适用原因：资源竞争复杂，模型假设不成立
  • 替代方案：考虑租户隔离和资源配额管理
